# Check Later — 프로젝트 기획서

> 일상에서 흩어지는 정보(영상, 글, 캡처, 메모)를 한 곳에 모으고, 잊지 않고 다시 찾을 수 있는 개인용 보관소.

---

## 1. 문제 정의

### 현재 패턴
- 유튜브에서 "나중에 봐야지" 하는 영상 → 캡처해서 갤러리에
- 개발자 커뮤니티/오픈톡에서 좋은 팁 → 카카오톡 "나에게 보내기"로 짧게 메모
- 깊이 정리할 때 → Notion에 글로

### 문제점
- **입력은 빠른데 조회·정리가 안 됨** → 결국 잊혀짐
- 캡처는 갤러리에서 다시 못 찾고, 카톡 메모는 검색되지만 형태/주제 구분 없이 시간순으로만 쌓여서 묻힘
- Notion은 진입 마찰이 커서 "빠른 캡처"용으로는 안 맞음

### 핵심 가설
> **저장이 빠른 도구는 이미 있다(카톡). 부족한 건 "잘 찾아지는 구조"다.**
> 그래서 이 프로젝트는 _카톡만큼 빠른 저장_ + _카톡에 없는 분류·조회_ 두 가지를 동시에 만족해야 의미가 있다.

---

## 2. 목표

- **저장 마찰을 카톡 나에게 보내기 수준으로 낮춘다** (Web Share Target API로 외부 앱의 "공유" 메뉴에서 바로 던지기)
- **3축 분류만으로 끝낸다** (형태 / 상태 / 날짜) — 자유 태그는 결국 안 쓰게 됨
- **모바일/PC 어디서나 같은 데이터** (클라우드 동기화)
- **혼자 쓰기 시작하지만, 다른 사람도 회원가입하면 쓸 수 있는 구조** (Supabase Auth + RLS)

---

## 3. 범위

### MVP 포함
- 회원가입/로그인 (이메일 또는 구글)
- 항목 저장: 텍스트 메모 / URL / 이미지(캡처) 중 하나 이상
- 분류: 형태(영상·글·캡처·메모) + 상태(안 봤음·봤음·보관) + 날짜(최근순)
- 목록 조회 + 필터 + 정렬
- 상태 변경 (안 봤음 → 봤음 → 보관)
- PWA 홈화면 설치 + Web Share Target

### MVP 제외 (의도적으로 자르는 것들)
| 제외 | 이유 |
|---|---|
| 능동 알림/리마인더 | 사용자 합의 — 알림 없이 수동 조회만 |
| 자유 태그 / AI 자동 분류 | 입력 마찰 증가, 결국 안 쓰게 됨 |
| 디스코드/카카오 봇 | V2 후보. PWA가 어차피 필요하므로 봇은 추가 부담 |
| 사용자 간 공유 | 개인 도구 정체성 유지 |

## 4. 기술 스택 및 선택 이유

> 이 섹션은 "왜 다른 후보가 아닌가"를 함께 기록해 향후 의사결정 근거로 남긴다.

### 프론트엔드: React + Vite (Next.js 아님)
- **왜 Next.js가 아닌가:** Next.js는 SSR/SSG, 서버 컴포넌트, 파일 기반 라우팅 등 풀스택 프레임워크다. 이 프로젝트는 백엔드를 Supabase에 위임하므로 서버 측 렌더링이 불필요하다. PWA는 본질적으로 SPA(Single Page App)에 가까워서 CSR이 더 자연스럽다. Next.js를 쓰면 빌드 복잡도와 학습 곡선이 늘어나는 만큼의 이득이 없다.
- **왜 Vite인가:** 개발 서버가 빠르고, `vite-plugin-pwa`가 잘 정착돼 있어 Service Worker / manifest / Web Share Target 설정이 간단하다.

### 언어: TypeScript
- 런타임 에러를 컴파일 시점에 잡음
- Supabase가 DB 스키마에서 타입을 자동 생성해주므로 프론트-DB 타입 정합성이 보장됨

### 스타일: Tailwind CSS
- 별도 CSS 파일 왔다 갔다 안 해도 됨 (모바일 우선 반응형이 클래스 prefix로 쉬움)
- 빠른 프로토타이핑에 최적

### 상태 관리: TanStack Query + Zustand (Redux 아님)
- **왜 Redux가 아닌가:** 보일러플레이트가 많고, 1인 사이드 프로젝트에 과하다. Redux Toolkit으로 가벼워졌지만 여전히 무겁다.
- **TanStack Query (서버 상태):** Supabase에서 가져오는 데이터(목록, 상세 등)의 캐싱·리페치·낙관적 업데이트를 자동 처리. "서버 상태"는 본질적으로 클라이언트 상태와 다르고, 이걸 Redux로 다루면 미들웨어를 직접 짜야 한다.
- **Zustand (UI 상태):** 필터 선택값, 모달 열림/닫힘 같은 순수 UI 상태. 작고 단순하며 Provider 래핑이 필요 없다.

### 백엔드/DB/스토리지/인증: Supabase (자체 서버·Firebase 아님)
- **왜 자체 서버(Express/Nest)가 아닌가:** 인증, DB 마이그레이션, 이미지 업로드, 권한 검사를 다 직접 구현하면 1인 사이드 프로젝트가 끝나지 않는다.
- **왜 Firebase가 아닌가:** Firebase도 잘 맞지만 Firestore가 NoSQL이라 "형태 + 상태 + 날짜로 필터" 같은 복합 쿼리에 복합 인덱스를 직접 만들어야 한다. Supabase는 Postgres라 SQL로 직관적이다.
- **왜 Supabase인가:**
  - Postgres + Storage + Auth + Realtime을 한 번에
  - **Row Level Security(RLS)** 로 "다른 사용자 데이터 조회 불가"를 DB 차원에서 강제 → 다중 사용자 안전
  - 무료 티어 (월 5만 MAU, 500MB DB, 1GB Storage) — 본인 + 친구 몇 명 충분

### 배포: Vercel
- React/Vite 앱 무료 배포, 자동 HTTPS (PWA는 HTTPS 필수)
- GitHub 푸시 시 자동 빌드/배포

### 입력 채널: PWA (디스코드 봇·카카오 API 아님)

이 결정은 가장 오래 고민한 부분이라 별도로 기록한다.

#### 처음 떠올린 후보들
- **카카오톡 API:** 본인이 본인에게/특정 채널에 보내는 것 정도만 제한적으로 가능. 외부 서비스가 카톡 메시지를 받아 처리하는 워크플로우는 비즈니스 채널 + 사용자 동의 흐름이 복잡하고, 개인 사이드 프로젝트가 정식 절차를 밟기 어렵다. 사실상 불가능.
- **디스코드 봇:** 이미 설치된 앱 + 익숙한 UX라는 강점이 분명히 있다. "디스코드 열고 채팅 치면 끝" — 입력 마찰은 거의 0.

#### 그럼에도 PWA 단독으로 결정한 이유
1. **조회는 결국 봇으로 안 된다.** 분류·필터·상태 변경은 어차피 별도 UI가 필요하다. 즉 PWA는 어떤 선택을 해도 만들어야 한다. 봇은 "추가" 인프라다.
2. **PWA = 모바일에서 디스코드만큼 빠를 수 있다.** 홈화면 설치 시 네이티브 앱과 동일한 진입 UX(아이콘 한 번 탭 → 풀스크린 앱). 거기에 **Web Share Target API**로 등록하면 유튜브 등 외부 앱의 "공유" 메뉴에 우리 앱이 후보로 뜨고, URL·이미지가 자동으로 들어온다. 디스코드보다 한 단계 적은 탭으로 끝난다.
3. **분류 메타데이터 입력이 자연스럽다.** 디스코드 채팅에는 형태/상태 같은 메타데이터를 어떻게 줄지(이모지 반응? 슬래시 커맨드? 채널 분리?) 별도 설계가 필요한데, PWA에서는 드롭다운/버튼으로 자연스럽다.
4. **인프라 단순.** 봇 서버를 추가하면 Supabase + 봇 호스팅 두 곳을 관리해야 하고, 봇이 죽으면 입력이 끊긴다.

#### V2 후보로 남기는 이유
실제로 써보고 "그래도 디스코드 채팅이 더 빠르더라"고 느끼면 디스코드 봇을 입력 채널로 추가하면 된다. 봇은 Supabase에 쓰기만 하면 되므로 데이터 모델은 그대로 재사용된다. 지금 미리 만들지 않는 것뿐.

## 5. 데이터 모델

### items 테이블

| 컬럼 | 타입 | 비고 |
|---|---|---|
| `id` | uuid | PK |
| `user_id` | uuid | FK → `auth.users`, RLS 키 |
| `type` | enum | `'video' \| 'article' \| 'screenshot' \| 'memo'` |
| `status` | enum | `'pending' \| 'reviewed' \| 'archived'` |
| `title` | text | URL이면 og:title 자동 추출, 없으면 사용자 입력 |
| `memo` | text | 사용자 짧은 메모 (nullable) |
| `url` | text | nullable |
| `image_path` | text | Supabase Storage 경로 (nullable) |
| `created_at` | timestamptz | 기본값 now() |
| `updated_at` | timestamptz | 기본값 now() |

### RLS 정책
- SELECT/INSERT/UPDATE/DELETE: `user_id = auth.uid()` 만 허용

### Supabase Storage
- 버킷: `item-images` (private)
- 경로 규칙: `{user_id}/{item_id}.{ext}`
- 접근: signed URL 발급

## 6. 화면 구조

### 라우트
| 경로 | 화면 |
|---|---|
| `/login` | 로그인 / 회원가입 |
| `/` | 메인 목록 (필터 + 카드 리스트) |
| `/new` | 새 항목 추가 (PWA Web Share 진입점도 여기) |
| `/items/:id` | 상세 보기 / 수정 |
| `/settings` | 로그아웃, PWA 설치 안내 |

### 메인 목록
- 상단: 형태 칩(영상·글·캡처·메모) + 상태 탭(안 봤음·봤음·보관) + 정렬 토글(최근순/오래된순)
- 본문: 카드 리스트 (썸네일 + 제목 + 메모 미리보기 + 형태/상태 뱃지)
- 우하단 FAB: `+` 버튼 → `/new`

### 새 항목 폼
- URL / 이미지 / 메모 중 하나라도 있으면 저장 가능
- 형태 자동 추정 (URL → video 또는 article / 이미지만 → screenshot / 둘 다 없음 → memo) — 사용자가 변경 가능
- 상태 기본값 `pending`

## 7. 구현 마일스톤

각 마일스톤은 "검증 가능한 결과"로 끝낸다.

### M0: 프로젝트 세팅
- `npm create vite@latest` (React + TS)
- Tailwind, ESLint, Prettier 설정
- Supabase 프로젝트 생성, `.env.local`에 키 저장
- Git init
- **검증:** `npm run dev` → 빈 화면 뜸

### M1: 인증 + 기본 CRUD
- Supabase Auth 연동 (이메일 또는 구글)
- `items` 테이블 + RLS 정책 작성 (Supabase SQL Editor)
- 로그인 후 목록 조회 (빈 상태 UI 포함)
- 새 항목 추가 (텍스트 메모만)
- **검증:** 회원가입 → 메모 추가 → 새로고침 후에도 목록에 남아 있음

### M2: 이미지 + URL
- Supabase Storage `item-images` 버킷 + 정책
- 이미지 업로드 (파일 선택 / 드래그)
- URL 입력 시 메타데이터 자동 추출 (og:title, og:image) — 서버리스 함수 또는 oEmbed
- **검증:** 유튜브 URL 붙여넣으면 제목·썸네일 자동 입력. 이미지 업로드 후 목록에서 보임.

### M3: 분류 + 필터
- 형태/상태 뱃지 표시
- 형태 칩 + 상태 탭 필터 작동
- 상태 변경 액션 (안 봤음 → 봤음 → 보관)
- **검증:** 10개쯤 더미 데이터 만들고 각 필터로 좁혀짐을 확인

### M4: PWA + 공유
- `vite-plugin-pwa` 설정 (manifest, Service Worker, 아이콘)
- Web Share Target API 등록
- **검증:** 모바일 크롬에서 사이트 접속 → "홈 화면에 추가" → 아이콘 탭으로 풀스크린 진입 → 유튜브 앱에서 공유 메뉴에 우리 앱 뜸 → URL/이미지 자동 전달

### M5: 배포
- Vercel에 연결 (GitHub 푸시 자동 배포)
- 환경 변수 설정 (Supabase URL/Key)
- 모바일 실기기 테스트
- **검증:** 폰에서 도메인 접속 → 홈화면 설치 → 유튜브 공유 → 저장 → PC 브라우저에서 같은 항목 조회됨

### V2 후보 (실사용 후 결정)
- 디스코드 봇 입력 채널 (입력 마찰이 여전히 신경 쓰일 때만)
- AI 자동 요약/분류 (Claude API 등)
- 자유 태그
- 능동 리마인더 (망각곡선 기반 복습 알림)
- 사용자 간 공유

## 8. 열린 질문 (구현 중 결정)

- 인증 방식: 이메일/패스워드 vs 구글 OAuth — 본인 편의로 선택
- URL 메타데이터 추출을 클라이언트(CORS 우회 필요)에서 할지 Supabase Edge Function에서 할지 — M2 시점에 결정
- 이미지 압축/리사이즈를 클라이언트에서 미리 할지 — 무료 티어 1GB 한계 고려해 추후 결정
